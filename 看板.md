>>
>> - `external-skills/` 只做来源追溯，不是正式运行入口。

>> ### 总控运行架构
>>
>> | 层级 | 能力 | 使用方式 |
>> |---|---|---|
>> | 总控入口 | AI 原生交付总控 | 先判断 `CODE_DELIVERY`、`DOCUMENT_DELIVERY` 或 `MIXED_DELIVERY`，再选择应用并控制授权 |
>> | 代码交付底座 | 全栈交付、运行态检查、严格 MSCE、Git 门禁 | 交付可运行、可验证、可提交的代码结果 |
>> | 文档交付底座 | 文档编排、证据、内容、格式、安全、交付包与归档门禁 | 交付可阅读、可追溯、可分发的文档结果 |
>> | 上层应用 | UI/UX 改版、产品测试、需求与 PRD、复杂方案专题会 | 在总控基础上处理具体业务场景，并自动继承所需底座 |
>>
>> > [!important] 双路线汇合规则
>> > 只交付代码走代码底座，只交付文档走文档底座；两者同时交付时分别维护候选范围。只有 `MSCE_SUBMISSION_GATE: PASS` 与 `DOCUMENT_DELIVERY_GATE: PASS` 都有效，并且版本、接口名称、验收结论等共享事实一致，才能标记 `MIXED_DELIVERY_GATE: PASS`。
>>
>> ### 最常用快捷命令
>>
>> > [!skill-command] 01 · 启动 AI 原生交付总控
>> > **适用：**不知道该选哪个应用，或任务会跨需求、设计、开发、测试与交付。
>> >
>> > ```text
>> > 启动 AI 原生交付总控，处理【具体任务】。
>> > 目标是【期望结果】；最终交付【代码/文档/代码+文档】；保留【现有业务/页面/接口】不变；验收标准是【标准】。
>> > 先判断 CODE_DELIVERY、DOCUMENT_DELIVERY 或 MIXED_DELIVERY，再选择对应底座。
>> > 请判断应调用 UI/UX 改版、产品测试、需求与 PRD、复杂方案专题会中的哪个上层应用。
>> > 代码结果自动组合全栈、运行态、严格 MSCE 与 Git 门禁；文档结果自动组合证据、内容、格式、安全与交付包门禁；混合结果必须双门禁汇合。
>> > git add、commit、push 或 PR 必须分别等待我确认。
>> > ```
>>
>> > [!skill-command] 02 · UI/UX 改版
>> > **适用：**后台、工作台、表单、认证页和业务页面改版。
>> >
>> > ```text
>> > 使用 Oristrat 产品 UI 总控 redesign【页面/组件】。
>> > 目标用户是【角色】，首要任务是【用户要完成的动作】。
>> > 保留现有业务、路由、权限、文案意图和数据契约，不引入新的 UI 库。
>> > 参考【现有页面/截图】，按需使用 ui-ux-pro-max 候选和 Hallmark 审计，最后进行真实界面验收。
>> > 只交付审计/设计说明走文档底座；只改代码走代码底座；实现与正式设计文档同时交付时走 MIXED_DELIVERY。
>> > ```
>>
>> > [!skill-command] 03 · 产品测试
>> > **适用：**功能验收、回归、缺陷复测和交付证据整理。
>> >
>> > ```text
>> > 使用 product-testing 对【URL/功能】执行【验收/回归/复测】。
>> > 账号与前置条件为【测试条件】，重点覆盖【关键业务路径】。
>> > 同时检查正常、空、错误、禁用、关闭重开和真实接口返回，并保留失败证据。
>> > HTTP 200 只代表连通，最终必须验证业务结果和页面反馈。
>> > 只做运行验证不强制生成文档；正式测试报告/证据包走文档底座；修复代码并交付报告时走 MIXED_DELIVERY。
>> > ```
>>
>> > [!skill-command] 04 · 需求与 PRD
>> > **适用：**把模糊想法整理为研发可以直接执行的规格。
>> >
>> > ```text
>> > 使用 product-design-management 把【需求或想法】整理成研发可执行规格。
>> > 用户角色是【角色】，核心问题是【问题】，期望结果是【业务结果】。
>> > 输出业务流程、状态、失败路径、权限、验收标准、非目标和待确认项。
>> > 不虚构接口、指标或业务规则；缺失信息单独标记。
>> > 只交付 PRD 走 DOCUMENT_DELIVERY；同时继续开发时切换为 MIXED_DELIVERY。
>> > ```
>>
>> > [!skill-command] 05 · 复杂方案专题会
>> > **适用：**存在多个方案、跨团队取舍或重大风险的决策。
>> >
>> > ```text
>> > 使用 task-council 针对【议题】召开专题会。
>> > 决策目标是【要做出的决定】，必须回答【关键问题】。
>> > 让【产品/研发/测试/安全/业务】席位独立提交证据、方案、风险和反对意见。
>> > 保留未解决分歧，最后输出决策、拒绝方案、责任人和下一步验证。
>> > 会议结论走 DOCUMENT_DELIVERY；若决策同时进入代码实施则走 MIXED_DELIVERY。
>> > ```

>> ### 什么是好样本
>>
>> | 维度 | 好样本 | 不足样本 |
>> |---|---|---|
>> | 对象 | 明确页面、文件、URL 或业务流程 | “这个东西” |
>> | 结果 | 描述用户最终能完成什么 | “优化一下” |
>> | 边界 | 指定保留项、非目标和不能改的部分 | 默认全仓修改 |
>> | 验收 | 写清正常、错误、恢复和真实接口标准 | 只要求 build 通过 |
>> | 授权 | 明确哪些动作可以直接做、哪些逐轮确认 | 用一句“可以”批准全部后续动作 |
>>

>> ### 双路线判断样本
>>
>> | 用户结果 | 总控分类 | 自动组合的底座 |
>> |---|---|---|
>> | 完成忘记密码页面并准备提交 | `CODE_DELIVERY` | 全栈实现、运行验收、严格 MSCE、Git 分轮 |
>> | 输出找回密码需求 PRD | `DOCUMENT_DELIVERY` | 来源证据、内容复核、格式与安全门禁 |
>> | 完成功能并交付正式测试报告 | `MIXED_DELIVERY` | 代码与文档两条路线独立通过后汇合 |
