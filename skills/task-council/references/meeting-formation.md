# 会议组建

## 先确认主旨，再选择参会者

先形成会议大纲，明确核心决策、决策层级、范围内外、成功标准、决策负责人和预期交付。不要从角色库中凑人数，也不要在主旨存在实质歧义时直接模拟讨论。

出现以下情况时，先向用户展示大纲并等待确认：

- “多个方案”的含义或所属决策层级不明确。
- 相邻议题可能改变会议范围或候选机制。
- 决策高成本、不可逆、跨团队或具有重大风险。
- 用户要求正式专题会、反复辩论或真实多 Agent。

清晰、低风险且可逆的任务可以标记 `PROVISIONAL`，展示关键假设后继续。模拟会议只免除真实派发确认，不免除主旨对齐。

## 会议大纲确认模板

```text
会议：<议题>
核心决策：<一句话，不包含预设解法>
决策层级：<战略 / 架构 / 机制 / 参数 / 执行>
决策负责人：<人或角色>
成功标准与硬约束：<内容>
本次必须解决：<内容>
本次明确不解决：<内容>
另开专题：<相邻议题>
预期交付：<类型>
待用户确认或纠正：<1-3 个真正改变会议方向的问题>
大纲状态：AGENDA_CONFIRMED / PROVISIONAL / NEED_CLARIFICATION
```

只询问会实质改变决策层级、范围、候选机制或交付的问题。优先提交一份可修改的草案，不要把会议准备变成冗长问卷。

## 会议模式

| 模式 | 建议规模 | 适用情况 | 最小交付 |
|---|---:|---|---|
| 快速碰头 | 2-3 个角色 | 低风险、可逆或早期方向 | 建议、主要异议、下一项验证 |
| 标准专题会 | 3-5 个角色 | 跨职能方案或重要权衡 | 证据、独立方案、异议、行动计划 |
| 重大决策 | 4-5 个角色；只为关键角色增加席位 | 不可逆、高成本、安全、隐私或战略决策 | 完整决策记录、保护措施、停止或回滚条件 |
| 复盘 | 原角色加独立审计或反方 | 目标落空、事故或反复分歧 | 因果假设、证据缺口、纠正行动 |

## 外部调研专题会

当用户要求仓库、服务商、免费试用或市场资料时，先完成 `research-discovery.md`，再按下表组会：

| 席位 | 独立问题 | 证据边界 | 禁止越权 |
|---|---|---|---|
| 需求澄清席 | 本次到底筛选什么，什么不在范围内？ | 用户确认的卡片与项目约束 | 不替用户假定技术方案。 |
| 开源侦察席 | 哪些仓库满足可部署、许可和维护要求？ | 官方仓库、许可证、部署/发布资料 | 不把通用技术组件扩写成架构。 |
| 服务商侦察席 | 哪些产品满足目标能力和已证实的免费条件？ | 官方产品、价格和试用资料 | 不把销售申请称为免费试用。 |
| 红队席 | 哪些候选因证据、许可、数据边界或免费条件应排除？ | 原始反例和硬约束 | 不用偏好替代排除证据。 |

`SIMULATED` 只能输出结构化分析和明确的证据限制。只有上述问题具有独立证据边界且真实 Agent 已获确认时，才可称为真实多视角专题会。

## 角色选择矩阵

| 任务信号 | 必选或优先角色 |
|---|---|
| 目标、价值或范围存在分歧 | 任务发起/业务、用户与体验、反方 |
| 技术、架构、数据或可行性未知 | 专业领域、执行与运营、反方 |
| 外部竞争、行业变化或政策影响显著 | 市场与外部情报、专业领域、风险与合规 |
| 预算、采购、资源抢占或长期成本关键 | 财务与资源、执行与运营、反方 |
| 用户数据、权限、安全、合同或法规相关 | 风险与合规、专业领域、执行与运营 |
| 需要真正落地或跨团队协作 | 执行与运营、任务发起/业务、专业领域 |

主席负责控制轮次和收敛，秘书负责事实台账、方案合并、纪要和行动项。两者不作为独立表决席位。决策负责人可以兼任业务发起人，但不得兼任独立反方席。

## 席位配置

每个咨询角色按需要配置互补席位：

- `倡议席`：阐明机会、收益、成功路径和应保留的选择。
- `审慎席`：识别反例、激励错配、风险、证据缺口和不应做的事。
- `落地席`：列出资源、依赖、顺序、验收、运营成本和回滚办法。

优先选择 `倡议 + 审慎`。只有执行方式会改变结论时才加入 `落地席`。每席必须写清独立问题、证据边界和禁止越权范围；只能重复时减少席位。

## 真实 Agent 派发确认

会议大纲确认后，若拟使用真实 Agent，再向用户展示：

```text
运行模式：REAL_MULTI_AGENT
参会 Agent：<角色 — 独立工作合同> ...
证据/工具边界：<按 Agent 列出>
预期轮次：<独立提案 / 交叉质询 / 修订 / 独立验证>
预计工作量与外部成本：<估计>
派发状态：DISPATCH_CONFIRMED / NOT_CONFIRMED
```

只有 `DISPATCH_CONFIRMED` 后才派发。未确认或工具不可用时，使用 `SIMULATED` 或 `NEED_EVIDENCE`，并说明限制。
