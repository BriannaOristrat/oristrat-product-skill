# 方案结构与独立性

## 使用前提

仅在需求发现已确认“本次要比较架构、机制、参数或执行方案”时使用本文件。若用户要的是开源仓库或服务商调研清单，先使用 `research-discovery.md` 与外部调研席位；仓库和服务商是候选来源，不应被改写成架构方案或参数情景。

## 决策层级

先判断会议要比较哪一层，避免把不同层级内容混成并列方案。

| 层级 | 核心问题 | 典型输出 |
|---|---|---|
| 战略 | 为什么做、为谁做、追求什么结果 | 方向、定位、取舍原则 |
| 架构 | 系统由哪些部分组成，边界与责任如何划分 | 架构路线、业务模式、治理结构 |
| 机制 | 核心规则如何运转，价值和风险如何流动 | 运行机制、交易机制、流程机制 |
| 参数 | 已选机制采用什么阈值、倍率、额度或档位 | 参数包、敏感性区间 |
| 执行 | 以什么顺序、资源和验证方式落地 | 路线图、实验、实施计划 |

参数不同不等于机制不同；执行顺序不同也不自动构成架构方案。

## 机制指纹

为每个候选填写以下结构。字段名称可按任务调整，但必须覆盖决策对象、运行方式和风险归属。

| 维度 | 要回答的问题 |
|---|---|
| 决策对象 | 方案作用于什么行为、资源、结果、权益或责任 |
| 对外单位或承诺 | 用户、客户或协作方看到什么单位、档位或承诺 |
| 内部核算或控制基准 | 后台依据成本、用量、结果、预算、容量还是固定规则 |
| 核心机制 | 固定、动态、预分配、事后结算、订阅、市场化、审批制或其他机制 |
| 发生时点 | 事前、事中、事后或周期性发生什么动作 |
| 风险归属 | 波动、失败、超额、异常与不可逆损失由谁承担 |
| 可预测性与可解释性 | 参与者能否事前理解结果，事后能否解释 |
| 治理与例外 | 上限、熔断、退款、回滚、审批、审计和例外如何处理 |

## 独立方案判定

候选满足以下任一条件，才可保留为独立方案：

- 至少两个机制指纹维度发生结构变化。
- 一个核心维度发生变化，并实质改变对外契约、责任边界、风险归属、治理方式或不可逆后果。
- 候选需要不同的能力体系、组织责任或验证方法才能成立。

以下变化默认合并为同一机制的参数情景：

- 只改变倍率、价格、折扣、阈值、权重、额度或容量。
- 只更换供应商、数据源、成本口径或实现组件，但对外机制和责任边界不变。
- 只叠加高峰、排队、风险、缓存、区域或其他修正系数。
- 只用“保守、标准、激进、基础、高级”等标签重新命名相同机制。

若不确定是否独立，执行替换测试：把候选 B 的参数代入候选 A 后，A 是否仍能表达 B 的运行方式和责任关系？若可以，B 通常只是 A 的参数变体。

## 生成与去重顺序

1. 按会议大纲列出可能的机制原型，不标记推荐顺序。
2. 为每个原型填写机制指纹。
3. 合并只存在参数差异的候选，并把参数情景放在所属机制下。
4. 删除不满足硬约束或没有证据支撑的伪选项。
5. 为剩余独立方案使用同一评价标准。
6. 在质询和修订后再次去重；允许回应 `MERGE_AS_VARIANT`。

不要为满足数量要求制造方案。若独立方案不足，报告原因、缺失证据和下一步探索问题。
