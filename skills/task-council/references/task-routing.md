# 任务路由与证据要求

## 路由原则

`task-council` 负责需求发现、会议主旨、组会、分配问题和收敛结论。专业工作优先交给相应自研 Skill、正式安装的系统/插件能力、项目资料或可验证工具。外部能力只能在需求发现确认后选择；它们是取得证据的工具，不定义调研范围，也不替代席位或验收。

| 任务类型 | 首选专业能力 | 最小证据 |
|---|---|---|
| 外部事实、竞品、政策、市场 | 正式网络调研能力；需要网络调研时使用 `agent-reach` | 一手或权威来源、日期、适用范围 |
| 产品、需求、路线图、PRD | `product-design-management` | 用户、目标、范围、验收标准 |
| 研发、架构、代码或交付 | 对应工程/测试 Skill 与项目规则 | 代码、设计约束、验证命令或测试证据 |
| UI、体验、信息架构 | UI/UX 或设计 Skill | 用户流程、状态、可用性或界面证据 |
| 数据、指标、商业分析 | 数据分析或策略 Skill | 指标口径、数据来源、时间范围、计算假设 |
| 风险、隐私、安全、合规 | 对应专业审查与权威材料 | 适用规则、影响范围、审批或阻断条件 |

## 外部调研的最低证据

| 调研线 | 必要证据 | 不能替代的内容 |
|---|---|---|
| 开源仓库 | 官方仓库、许可证、部署入口、维护证据、适用边界 | 星标、博客转述或模型名称。 |
| 服务商与免费试用 | 官方产品页、价格/试用页、免费分类、额度/期限、开通条件、数据边界 | “可联系销售”“页面有免费字样”或过期活动。 |
| 红队排除 | 触发的硬约束、原始来源、排除理由 | 主观不喜欢或多数意见。 |

## 证据台账

在首轮发言前建立以下台账：

| 编号 | 主张 | 分类 | 来源或验证动作 | 负责人 | 状态 |
|---|---|---|---|---|---|
| E-01 |  | FACT / ASSUMPTION / INFERENCE / UNKNOWN |  |  |  |

只有 `FACT` 可被表述为已确认。`ASSUMPTION` 和 `INFERENCE` 必须在最终决策中保留验证或风险说明；`UNKNOWN` 不得被讨论直接消除。外部来源还必须记录日期和适用范围。
