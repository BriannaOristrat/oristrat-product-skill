# 产品设计与产品管理工作流

## 入口判断

先判断用户要解决的是哪类问题：

| 类型 | 典型输入 | 主要产物 |
|---|---|---|
| 模糊想法 | 想做一个功能、体验不好、帮我规划 | 问题定义、用户场景、机会判断、下一步调研 |
| 需求分析/需求沉淀 | 帮我做需求分析、写 PRD、产品规范、功能说明 | 需求分析结论、PRD、范围、验收标准、指标、风险 |
| 设计推进 | 设计方案、用户流程、原型、UI 评审 | 设计 brief、流程图、状态清单、交互说明、评审意见 |
| 决策排序 | roadmap、优先级、取舍 | 评分矩阵、推荐顺序、取舍理由 |
| 研发交付 | 拆任务、ticket、交付计划 | Epic、用户故事、任务拆解、Definition of Done |

## 核心流程

1. 明确背景：业务目标、用户对象、当前问题、已有证据、约束。
2. 定义问题：写出用户痛点、发生场景、频率/影响、为什么现在要解决。
3. 选择路径：研究、需求分析、PRD、设计、路线图、任务拆解或评审。
4. 形成产物：按任务类型套用 `templates-and-gates.md`。
5. 过门禁：检查范围、指标、验收、风险、依赖和未决问题。

## 产品发现

输出至少包含：

- 目标用户和关键场景。
- 用户任务或 Jobs To Be Done。
- 当前替代方案和摩擦点。
- 问题严重度、频率、商业影响。
- 需要验证的假设。
- 推荐下一步：访谈、数据分析、原型、实验或直接进入 PRD。

## 需求分析、PRD 与规范

需求分析和 PRD 必须回答：

- 为什么做：业务目标、用户问题、机会判断。
- 做什么：范围、流程、功能、状态、边界。
- 怎么算成：成功指标、验收标准、测试样例。
- 不能做什么：非目标、延期项、风险、依赖。

如果需求会进入研发，必须把验收标准写成可测试行为。

## 产品设计

设计工作必须覆盖：

- 目标用户、核心任务和入口路径。
- 主流程、异常流程、空状态、加载状态、错误状态。
- 信息架构和页面/组件层级。
- 交互反馈、表单校验、权限和边界状态。
- 设计系统复用情况和新增组件影响。
- 可访问性、响应式、国际化或本地化要求。

## 路线图与优先级

优先级建议要显式说明评分依据。常用维度：

- 用户价值。
- 商业价值。
- 战略匹配。
- 证据强度。
- 实现复杂度。
- 风险和依赖。
- 时间敏感性。

不要只给排序，要说明为什么排在前面、为什么暂缓。

## 交付拆解

从 PRD 到研发任务时，按以下层级拆：

1. Epic：一个完整用户价值闭环。
2. Story：用户可感知的行为变化。
3. Task：工程实现、数据、设计、测试、文档等可执行项。
4. Acceptance Criteria：每个 Story 的可验证完成条件。
5. Definition of Done：测试、文档、埋点、回滚、上线条件。

## 评审输出

评审类输出按严重度组织：

- P0：会导致目标失败、严重误导、数据/权限风险。
- P1：明显影响主流程效率、理解或转化。
- P2：局部体验、文案、视觉或一致性问题。
- P3：优化建议或后续增强。

每条建议要包含位置、问题、影响、建议改法。
