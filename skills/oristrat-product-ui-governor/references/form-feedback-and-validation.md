# 表单、反馈与恢复门禁

用于登录、找回密码、验证码、用户管理、设置、弹窗表单和其他异步提交界面。

## 1. 事实源复用

实现前建立复用映射：

| 需求 | 首选事实源 | 必查内容 |
|---|---|---|
| 登录字段与样式 | 当前 login 页面和组件 | label、placeholder、高度、间距、图标、企业代码、手机号传递 |
| 密码规则 | 用户管理实际校验函数/Schema | 长度、字符类型、错误文案、确认密码 |
| Toast | 当前通知组件和语义 Token | success/warning/error、时长、aria-live |
| API | Logic/Data/Server contract | 字段名、business success、错误码、超时与重试 |

优先抽取或复用共享校验源；如果暂时不能复用，记录重复规则和后续漂移风险。不得凭记忆重写正则或接口字段。

## 2. 错误与反馈位置

- 单字段格式错误、必填错误、密码不一致：放在对应输入框下方。
- 多字段组合错误、服务端业务失败：放在表单级反馈或 Toast，并尽可能定位字段。
- 成功结果：success/绿色 Toast。
- 时效、安全和提醒信息：warning/黄色 Toast。
- 网络、服务端和不可恢复错误：error/红色 Toast，并提供恢复动作。
- Toast 不抢焦点；使用合适的 `aria-live`。颜色不是唯一的信息载体。

## 3. 提交状态

- 缺少客户端可确定的必填信息或字段格式错误时，主按钮 `disabled`。
- 提交中禁用重复提交并显示进行中状态。
- 不要把“是否点击过获取验证码”当成验证码是否有效的事实源；最终由服务端验证码校验决定。
- 客户端可以提示验证码请求状态，但不得用本地布尔值阻断一个服务端可能接受的有效验证码。

## 4. 验证码与中断恢复

至少覆盖：

```text
未获取 / 已获取 / 倒计时 / 关闭后重开
旧验证码仍有效 / 已过期 / 错误 / 重发
网络失败 / 服务端业务失败 / 成功
```

- 明确哪些状态应在关闭弹窗后保留：登录输入的手机号/企业代码、用户已收到且仍可能有效的验证码，不应因本地弹窗关闭被误判无效。
- 明确哪些状态应重置：密码明文、提交中状态、瞬时错误；按安全和产品要求处理。
- 重新打开后允许直接提交一个服务端仍认可的验证码，不显示“请先获取验证码”这种客户端臆断。
- 重发后的旧验证码是否失效由后端合同决定，前端不得自行假设。

## 5. 密码字段

- 新密码和确认密码都提供一致的小眼睛显示/隐藏控件。
- 小眼睛有可访问名称、焦点状态和一致图标。
- 密码规则提示放在输入区域附近；错误在用户完成输入或提交后出现，避免每次按键制造噪声。
- 两次密码不一致必须有就近错误，并使提交按钮不可用。

## 6. Browser Acceptance

真实点击至少覆盖：

1. 从登录页携带手机号和企业代码进入。
2. 企业代码为空时才要求补填。
3. 手机号或邮箱错误、密码不一致的 inline 提示。
4. 获取验证码、关闭、重新打开、使用仍有效验证码提交。
5. 未在本次弹窗点击获取验证码但服务端验证码正确的提交。
6. warning/success/error Toast 的文字和语义颜色。
7. 成功后关闭弹窗且不显示冗余成功状态页面。
8. 有授权测试数据时完成真实密码重置并用新密码登录。

只有前六类 UI 行为和真实后端业务成功都验证后，端到端状态才是 `PASS`；缺少有效测试手机号或验证码时标记 `PARTIAL` 或 `BLOCKED`。
