# 页面类型与结构路由

按用户任务选择结构。结构差异必须服务业务，不为视觉新鲜度而变化。

## 路由表

| 页面类型 | 第一视口任务 | 优先结构 | 避免 |
|---|---|---|---|
| 审批/运营工作队列 | 找到异常并批量处理 | 命令栏 + 筛选 + 表格/列表 + 快速详情 | 指标卡占满首屏、每条记录单独卡片 |
| CRM 列表与跟进 | 定位客户并完成下一动作 | 保存视图 + 列表/看板 + 详情抽屉/分栏 | 无状态流转的通用 Dashboard |
| 财务/报销/发票 | 审核金额、凭证和风险 | 高密度表格 + 风险标记 + 审批上下文 | 只展示图表、不提供处理入口 |
| 租户/权限/系统管理 | 查找对象并安全变更 | 搜索筛选 + 表格 + 详情页/抽屉 + 确认门禁 | 把复杂配置塞进大 Modal |
| 分析 Dashboard | 发现变化并下钻原因 | 结论/告警 + 关键指标 + 趋势 + 下钻入口 | 无决策含义的装饰图表 |
| 设置页 | 理解分组并安全保存 | 分组导航 + 表单区 + 就近校验/保存 | 一个超长表单或多个嵌套卡片 |
| 创建/审批长流程 | 完成多步骤高风险任务 | 页面流程或 Wizard + 草稿 + 校验摘要 | 超大 Modal、丢失上下文 |
| 官网/营销页 | 理解价值并采取行动 | 按产品与受众选择叙事宏结构 | 固定 Hero → 三卡片 → CTA 模板 |
| 单个组件 | 覆盖行为、状态和系统一致性 | 组件 + 状态预览/测试 | 为单组件引入页面级主题和 IA |

## 范围规则

- `component`：不选择页面宏结构，不新建导航和 Footer；检查适用状态与现有 Token。
- `page`：选择一种与任务匹配的结构，保留系统导航和跨页设计语言。
- `system`：先形成或更新 `design.md`，统一 Token、组件和布局规则，再逐页迁移；不得一次性推翻全部页面。

## 内部产品与营销页分流

内部 SaaS、CRM、财务、后台和运营页面以效率、密度、一致性和可恢复操作为先。禁止为了“反 AI 味”让同一产品中的页面采用互不兼容的主题。

营销页、产品介绍和品牌页面可以提高结构方差与视觉表达，但仍须使用真实内容、可访问交互和品牌事实。没有明确创意信号时，沿用项目设计系统，不自动创建新主题。

## 视口验证矩阵

| 类型 | 必测宽度/视口 |
|---|---|
| 内部产品 | `1024×720`、`1280×720`；高风险表格再测 `1440×900` |
| 官网/营销页 | `320`、`375`、`414`、`768`、`1280` 宽度 |
| 单个组件 | 组件容器最窄可用宽度、`320` 宽度页面、正常桌面容器 |

所有类型都检查：无意外水平滚动、中文按钮不竖排、长文本/数字不重叠、操作区可达、关键内容不被 sticky 元素遮挡。
