# Oristrat 视觉质量门禁

本门禁吸收 Hallmark 的预扫描、真实性、Token、结构多样性、状态和响应式思想，并按 Oristrat 内部产品场景重写。不要复制外部 Skill 的完整规则或强制主题轮换。

## 1. 真实性

- 不虚构指标、增长率、客户、评价、Logo、案例、截图、集成或 AI 置信度。
- 缺失数据使用明确占位、待确认标记或改用不依赖数字的结构。
- AI 输出标明来源、假设和不确定性；高风险动作提供确认、拒绝、撤销或人工复核。

## 2. 现有系统优先

- `design.md`、项目 Token、组件和全局样式优先于外部主题。
- 字体、颜色、间距、圆角、阴影和动效通过命名 Token 使用。
- 新值先进入系统 Token，再被组件引用；避免页面内散落 hex、RGB、OKLCH 或独立字体声明。
- 不覆盖全局样式入口，不删除 Tailwind、`@import` 或框架基础指令。

## 3. 结构与层级

- 页面结构由用户任务、对象和状态决定。
- 内部产品避免无意义的 Hero、装饰性指标卡、卡片套卡片和平均分配的模块网格。
- 营销页避免固定的居中标题、三功能卡和通用 CTA 节奏；不同结构必须来自内容差异。
- 首屏明确“现在要处理什么”和“下一步做什么”，无需先阅读多段说明。

## 4. 字体、颜色与图形

- 保持清晰的字号层级和合理行长；标题使用正常体，避免把斜体强调当作默认 AI 风格。
- 状态颜色表达业务含义，不仅用于装饰；文字和控件满足可读对比度。
- 不手绘假浏览器、假手机、假 IDE 或假代码窗口；使用真实截图或直接展示内容。
- 图标遵循同一图标族，常用工具可用图标，关键命令保留清晰文字。

## 5. 文案密度

- 删除解释常识控件、重复标题和描述界面本身的段落。
- 保留能防止真实错误、说明业务规则或解决歧义的辅助文案。
- 不出现持续堆叠的“标题 + 副标题 + 描述”；能由标签、状态、计数器和空状态表达时不用段落。

## 6. 交互状态

按组件和业务责任覆盖适用状态：

```text
default / hover / focus-visible / active / disabled
loading / empty / error / success / permission-denied / long-content
```

- 单个按钮不必独自承担业务 `error/success`，但所属表单或操作区域必须覆盖。
- 高风险提交防重复、可恢复，并清楚展示进行中和失败状态。
- 焦点可见，键盘顺序合理，触控目标足够，动效尊重 `prefers-reduced-motion`。
- 反馈颜色遵循语义：成功使用 success/绿色，提醒与安全提示使用 warning/黄色，错误使用 error/红色；同时提供文字或图标，不仅依赖颜色。
- 字段格式或两次输入不一致等可定位错误放在对应输入框下；跨字段、网络和操作结果使用 Toast 或表单级反馈。

## 7. 响应式和数据压力

- 按 `surface-routing.md` 的矩阵验证真实视口。
- 表格、筛选、批量操作和详情区在窄屏有明确降级策略，不靠隐藏关键操作解决。
- 检查超长中文、英文无空格字符串、大金额、多位小数、多标签、空数据和高行数。
- 按钮文字保持横向可读；必要时设置稳定宽度、`white-space: nowrap`、短标签或带 Tooltip 的图标按钮。

## 8. 动效克制

- 动效只解释状态变化、层级和空间关系。
- 密集工作台减少装饰性入场和 hover 动画；营销页也不得让动效阻碍阅读和操作。
- 移除后不损失信息的动效默认删除。

## 9. 交付前自检

以 `PASS`、`FAIL` 或 `NEEDS_WORK` 评估：

```text
业务匹配 / 信息层级 / 视觉执行 / 内容具体性 / 克制程度 / 结构合理性
```

任一项为 `FAIL` 不得宣布完成；`NEEDS_WORK` 必须写明证据、风险和下一步。评分只进入交付证据，不向生产 CSS 写设计工具印章。
