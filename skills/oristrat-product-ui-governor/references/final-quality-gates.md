# 最终 UI 质量门禁

交付前逐项标记 `PASS`、`FAIL`、`NEEDS_WORK` 或 `N/A`，并给出代码、截图、浏览器、测试输出或明确推理证据。

| 门禁 | 必查内容 | 失败条件 |
|---|---|---|
| 业务场景 | 角色、任务、对象、状态、权限、失败路径 | 页面只是通用模板，不能完成真实工作 |
| 信息架构 | 首页、导航、模块归属、列表/详情/设置边界 | 角色任务混杂、入口重复或命名误导 |
| 交互模型 | 表格、抽屉、Modal、页面流、反馈方式 | 所有操作都用同一种容器，复杂任务塞入大 Modal |
| 设计系统 | Token、字体、颜色、密度、组件一致性 | 页面私有样式扩散或覆盖现有系统 |
| 内容真实性 | 指标、客户、评价、Logo、截图、AI 来源 | 存在虚构内容或未标明的不确定输出 |
| 视觉质量 | 层级、首屏任务、结构、文案密度、克制 | 装饰性卡片/说明文案压过主要操作 |
| 状态与可访问性 | 焦点、键盘、对比度、触控、加载/空/错/权限 | 关键路径只有静态 happy path |
| 实现适配 | 路由、组件、权限、数据契约、全局样式 | 视觉改版破坏工程或业务边界 |
| 响应式与浏览器 | 指定视口、真实点击、溢出、长文本、按钮 | 只编译不检查真实页面，或关键布局破损 |
| 工程验证 | lint、typecheck、build、相关测试 | 高风险改动未验证或失败仍宣称完成 |
| 外部能力证据 | ui-ux-pro-max/Hallmark/frontend-design 的可用状态、模式、采用与拒绝内容 | 不可用却声称已使用，或外部候选覆盖内部业务与工程规则 |

## 模式差异

- `audit`：工程验证可标 `N/A`，但必须有足够的只读证据。
- `study`：实现、浏览器和工程验证可标 `N/A`；必须说明可借鉴与拒绝复制的边界。
- 设计方案：浏览器验证可标 `N/A`；必须提供可测试验收标准。
- 已实现改版：关键业务、响应式、真实交互和工程验证不得无理由标 `N/A`。

## 完成规则

- 任一关键项 `FAIL`：先修复再交付；无法修复时报告阻塞。
- 任一关键项 `NEEDS_WORK`：写明风险、责任人或下一动作，不能使用“已全部完成”。
- 最终回复只汇总重要门禁及证据，不粘贴整份检查清单。
